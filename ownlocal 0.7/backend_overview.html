01 / OVERVIEW

Project Overview

OwnLocal is a fintech platform that converts everyday local shopping into fractional equity ownership in neighbourhood shops.

🎯 What It Does
  • Customers shop at local Kolkata stores and earn equity tokens automatically (₹1 spent = 0.2 tokens).
  • Tokens accumulate into Bronze → Silver → Gold stakeholder tiers, unlocking perks and eventual revenue-share equity.
  • Merchants register via GSTIN verification, get a Merchant Suite dashboard showing stakeholders, reviews, and equity distribution.
  • The backend stores shop data (seeded from an Excel sheet), manages waitlist sign-ups, and handles merchant auth.
  • An offline verifier lets merchants manually push equity tokens to customer wallets via phone number.
📍 Target Market

Kolkata's 7 major commercial zones: Academic/Student Hub, South Kolkata Shopping Hub, Central Transit Hub, Historic/Gateway Hub, North Kolkata Residential Hub, IT & Tech Hub (Salt Lake Sector V), and Premium/Dining Hub (Park Street). Each has a distinct equity percentage configuration tuned to footfall and average spend.

🔑 Key Design Decisions
  • No user database: User state (equity, ledger, tokens) lives entirely in localStorage. This keeps the backend simple and avoids auth overhead.
  • SQLite over PostgreSQL: Zero-config, single-file DB. Perfect for hackathon demos and easy to reset.
  • Excel as seed source: Real Kolkata shop data is stored in Kolkata Hubs Excel Sheet Layout.xlsx and imported on startup.
  • CORS wildcard: allow_origins=["*"] lets the static HTML frontend connect from any origin without configuration.

04 / CORE LOGIC

Core Logic

Key business rules and processing algorithms in the backend.

🧮 Equity Calculation

When /transaction is called with a spend amount:

# Tokens earned: every ₹10 spent = 1 token tokens_earned = round(total / 10) # Equity earned: 2% of total spend (in ₹) equity_earned = round(total * 0.02, 2) # Example: ₹500 purchase → 50 tokens, ₹10.00 equity
🔐 Merchant Authentication
  • Registration: Shop name must be unique (checked against merchants table). GSTIN is stored uppercase. No hashing in this version — plain-text password for hackathon speed.
  • Login: Direct SQL match on shop_name + password. Returns 401 on mismatch. Session maintained in browser localStorage.
  • No JWT/sessions on server: The backend is stateless — any state is owned by the client. This is an intentional trade-off for hackathon simplicity.
🌱 Startup Seeding
  • On startup, create_tables() creates all 3 tables with IF NOT EXISTS — safe to call repeatedly.
  • seed_shops() checks if shops table is empty. If so, reads Kolkata Hubs Excel Sheet Layout.xlsx, maps each shop to a hub config, and inserts rows.
  • If Excel file is missing, seeding is silently skipped (shops table stays empty; frontend falls back to its own LOCATION_DATABASE).

07 / HOW TO RUN

How to Run the Backend

Get OwnLocal running locally in under 3 minutes.

📋 Prerequisites
  • Python 3.11 or higher
  • main.py and Kolkata Hubs Excel Sheet Layout.xlsx in the same folder
  • index.html can be opened directly in browser (no build step)
⚡ Quick Start
# 1. Create & activate virtual environment (recommended) python -m venv venv source venv/bin/activate # Windows: venv\Scripts\activate # 2. Install dependencies pip install fastapi uvicorn pandas openpyxl # 3. Start the server (hot-reload for dev) uvicorn main:app --reload --port 5000 # 4. Open the frontend open index.html # or double-click it in Finder/Explorer # API now live at: http://localhost:5000 # Swagger docs at: http://localhost:5000/docs
🔧 Port Configuration

The frontend auto-detects the backend URL:

// In index.html (line ~1521) const API_BASE = window.location.hostname === 'localhost' ? 'http://localhost:5000' // local dev : ''; // same-origin in production

⚠️ If you use a different port, update this constant in index.html. The backend itself uses port 5000 (or whatever --port you pass to uvicorn). Default FastAPI port is 8000 — make sure they match.

🗃️ Reset Database
# Delete the SQLite file to start fresh rm ownlocal.db # Restart server — tables + shops re-created automatically uvicorn main:app --reload --port 5000
08 / TESTING & DEBUGGING

Testing & Debugging

Tools and techniques for verifying the backend works correctly.

🔍 Using Swagger UI (/docs)

Visit http://localhost:5000/docs for the auto-generated interactive API. You can test every endpoint with real inputs directly in the browser — no Postman needed. This is your primary demo tool for judges.

🧪 cURL / Terminal Tests
# Health check curl http://localhost:5000/ # Fetch shops for Gariahat curl "http://localhost:5000/shops?pincode=700019" # Process a transaction curl -X POST http://localhost:5000/transaction \ -H "Content-Type: application/json" \ -d '{"shop_id": 1, "total": 500}' # Register a merchant curl -X POST http://localhost:5000/signup \ -H "Content-Type: application/json" \ -d '{"shop_name":"Test Shop","password":"demo123","gstin":"19AADCB2230M1ZP","pincode":"700019"}' # View all waitlist entries curl http://localhost:5000/waitlist
🐛 Common Issues
  • Port mismatch: Backend on 8000 but frontend expects 5000 → update API_BASE in index.html or add --port 5000 to uvicorn.
  • CORS error in browser: Should never happen (allow_origins=["*"]) but if it does, check that backend is actually running.
  • Excel not found: Place Kolkata Hubs Excel Sheet Layout.xlsx in the same directory as main.py. The app still runs without it — shops table is just empty.
  • "Module not found": Run pip install fastapi uvicorn pandas openpyxl inside your venv.
  • DB locked error: SQLite doesn't support concurrent writes well. Stop other connections before restarting server.
09 / HACKATHON Q&A

25 Hackathon Q&A

Clear, confident answers to the questions judges are most likely to ask during the demo.

Q01
What problem does OwnLocal actually solve?
Local shops lose customers to e-commerce giants. OwnLocal inverts this by making loyalty financially rewarding — customers earn fractional equity in shops they frequent, creating a stake in their success. Both sides win: shops get sticky loyalty, customers build neighbourhood wealth.
Q02
Why FastAPI over Flask or Django?
FastAPI gives us automatic Swagger UI at /docs, Pydantic validation, and async support — all with less boilerplate than Flask. It's production-grade and the fastest Python framework by benchmark. For a hackathon, the auto-docs alone save hours of documentation work.
Q03
Why SQLite instead of PostgreSQL or MongoDB?
SQLite is zero-config, ships with Python, and stores everything in a single file. For a hackathon demo, this means instant setup, easy reset (just delete the file), and no external service dependencies. The app is production-migratable — swapping in PostgreSQL requires only changing the connection string.
Q04
How is equity actually calculated?
Every ₹10 spent = 1 token. Tokens accumulate into tiers: Bronze at 500 (free monthly perk), Silver at 2,000 (5% discount + voting rights), Gold at 10,000 (legal revenue-share equity). The base equity percentage per shop ranges from 1.0% to 11.2% depending on the hub type and market demand.
Q05
Is this legally compliant? Can you give equity in a shop?
The Gold tier equity is designed as "sentimental/community equity" in the MVP — a loyalty contract, not a securities instrument. For production, it would use a tokenized ownership structure under India's SEBI sandbox or as a contractual profit-sharing agreement, which is legal. The MVP clearly demos the UX and economic model.
Q06
Where is user data stored? Do you have a user database?
By design, user data (equity balance, tokens, activity ledger) lives in localStorage on the client. This avoids a user auth database for the MVP, keeps the demo fast, and lets each judge see a fresh wallet. The /portfolio endpoint exists as a stub for future server-side user accounts.
Q07
How do you verify a transaction is real?
In the MVP, transactions are simulated in the UI. In production, verification would use UPI payment webhooks (Razorpay/PhonePe) — the /transaction endpoint would be called only after a confirmed payment. The GSTIN verification flow and offline mint feature show the production-ready path.
Q08
What is the Offline Verifier feature?
Merchants can manually log in-store purchases by entering the customer's phone number and bill amount. This pushes equity tokens directly to that customer's wallet, solving the "no app at checkout" problem. It's critical for Kolkata's informal retail sector where digital payments aren't universal.
Q09
How does the map work? Is it real-time?
We embed OpenStreetMap iframes with bounding boxes specific to each pincode area. The hotspot overlays are positioned with CSS top/left percentages. It's a real live map with no API key required. For production, we'd use Mapbox or Google Maps with real shop pins from the database.
Q10
How is the shop data populated?
On startup, the backend reads Kolkata Hubs Excel Sheet Layout.xlsx using Pandas. Each row maps to a hub type (Academic, IT, Premium, etc.), and the HUB_CONFIG dict assigns category, emoji, footfall, and equity percentage. This real Kolkata data makes the demo geographically authentic.
Q11
How does merchant authentication work?
Merchants register with shop name (unique ID) + password + GSTIN. Login does a direct SQL match. Sessions persist via localStorage flags. In production, we'd add bcrypt password hashing and JWT tokens — the architecture already supports this upgrade with minimal refactoring.
Q12
Can multiple shops have the same name?
No. The merchants table has a UNIQUE constraint on shop_name, and the backend returns a clear 400 error if a duplicate is attempted. This prevents accidental double-registration and is enforced at both the API validation layer and the database constraint level.
Q13
What is the token market feature?
In the Wealth Wallet (Portfolio section), users can list their earned tokens for sale to other community members. This creates a secondary market for hyper-local loyalty — someone who shopped at 8B Chai 200 times can sell their tokens to a new resident who wants instant Silver status. It's P2P equity trading at neighbourhood scale.
Q14
How does the AI insight work?
The aiInsight(shop) function is a rule-based engine (not an LLM call) that analyses shop rating, footfall, and category to generate investment-style insights like "Hidden gem!" or "Monitor before investing." It adds perceived intelligence without API latency. A real AI integration (Gemini/Claude) would be the next step.
Q15
What's the business model for OwnLocal?
Three revenue streams: (1) Transaction fee: 0.5-1% on verified purchases routed through OwnLocal. (2) Merchant SaaS: Monthly subscription for the Merchant Suite dashboard. (3) Token marketplace: Small % on secondary token sales. The equity model aligns all parties — OwnLocal wins only when local shops grow.
Q16
How does this scale beyond Kolkata?
The pincode-based architecture is city-agnostic. Scaling to Mumbai or Bangalore requires adding new pincodes to the LOCATION_DATABASE, uploading a new Excel file with local shops, and configuring hub types. The backend handles any pincode with the default config. A city-onboarding admin panel would be the next feature.
Q17
What's the waitlist for?
User-side app features (UPI bank linking, real equity certificates, portfolio analytics) aren't in the MVP. The waitlist captures demand from users who want the full product, demonstrates go-to-market traction, and gives us pincode data to prioritise which areas to launch in first.
Q18
How does OTP login work?
The OTP is simulated in the frontend for the demo — the 4-digit code is displayed in the OTP hint (in demo mode). In production, it would integrate with MSG91 or Twilio to send real SMS OTPs. The frontend flow is production-identical; only the SMS dispatch is mocked.
Q19
Is the app production-ready?
The architecture is production-ready in shape. What needs upgrading for launch: (1) Password hashing with bcrypt, (2) JWT auth tokens, (3) PostgreSQL migration, (4) Real UPI payment webhooks, (5) SMS OTP via MSG91. Each is a well-understood engineering task — the hard part (product design, UX, business logic) is already done.
Q20
How does the recommendation engine work?
The getRecommendedShops() function scores each shop as: rating × 20 + (footfall / 10) + bonus if high spender. Top 2 scoring shops get the "⭐ Recommended for you" badge. It's lightweight, runs in-browser, and can be upgraded to server-side ML recommendations using past transaction data.
Q21
What happens if the backend is down?
The frontend degrades gracefully. The apiFetch helper catches network errors and returns {error: "Cannot reach backend"}. The app still works for browsing shops, exploring the map, and simulating purchases — because the shop data and local state are managed client-side in localStorage.
Q22
What are the different stakeholder tiers and their benefits?
🥉 Bronze (500 tokens): Free monthly item (category-specific, e.g. coffee at a café). 🥈 Silver (2,000 tokens): 5% lifetime discount + voting rights on shop decisions. 🥇 Gold (10,000 tokens): Legal revenue-share equity certificate — actual fractional ownership in the shop's profits.
Q23
How does GSTIN verification work?
The frontend shows a verification spinner for 2 seconds before registering the merchant. In the MVP, GSTIN format is validated (must be exactly 15 characters, stored uppercase). Production would call the https://api.gst.gov.in public verification API to confirm the GSTIN is real and matches the business name.
Q24
What's the biggest technical risk and how do you mitigate it?
Equity fraud — users gaming the system with fake purchases. Mitigation: (1) UPI webhook verification ensures only real payments trigger equity. (2) GSTIN-verified merchants can't be faked. (3) Rate-limiting on the transaction endpoint. (4) Anomaly detection on spending patterns. The MVP's demo-mode simplicity doesn't expose these in production.
Q25
Why will OwnLocal win the hackathon?
Three reasons: (1) Real problem — local shop loyalty is broken in India, and OwnLocal fixes it with a genuinely novel economic mechanism. (2) Full-stack polish — working frontend + backend + real Kolkata data, not just a slide deck. (3) Clear path to market — GSTIN system, offline verifier, and merchant dashboard show we understand the real operational constraints of Indian retail.
